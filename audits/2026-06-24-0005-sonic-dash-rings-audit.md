- **Findings:**

### 1. Ring count pointer chain (from Cheat Engine screenshot)

6-level pointer chain starting from `SonicDash_R_Ring.exe+0x849FEC`:
```
[base+0x849FEC] -> +0x28 -> +0x28 -> +0x20 -> +0x2C -> +0x2C -> +0xE8 = rings
```
Implemented using `helpers->ReadIntPtr()` with `__try/__except` for safety.

### 2. TicketJackpot confirmed different counter

User confirmed `OutputTicketJackpot` at `+0x84A238` is tracking a different metric (not actual jackpot). Kept as-is pending further investigation.

### 3. Build toolset issue (VS2026)

Premake5 generates `.vcxproj` with `v141` (VS2017). Installed VS is 2026 (v18) with MSVC 14.51. Required toolset identifier is `v145`. Workaround: override `PlatformToolset` property at build time via MSBuild `/p:PlatformToolset=v145`. The `.vcxproj` is gitignored (regenerated by premake5).

### 4. Output enum expansion

`OutputRings` added before `NUM_OUTPUTS` in `Outputs.h`. Matching string `"Rings"` added to `s_outputNames[]` in `Outputs.cpp`. Index ordering preserved.

- **Risks:**
  - Pointer chain may break with game updates (common for Sega Nu titles)
  - Rings displayed as `UINT8` (0-255); if ring count exceeds 255, it will wrap — consider `UINT16` or `UINT32` in future
  - Build requires manual `/p:PlatformToolset=v145` override when using VS2026
- **Decisions:**
  - Used `__try/__except` around pointer chain to prevent crashes if chain becomes invalid
  - Cast rings to `UINT8` for consistency with other outputs (ticket, coin, etc.)
  - Deployed DLL to game root (replaced previous 6/23 build)
- **Implementation notes:**
  - Ring count sent via NetOutputs TCP as `"Rings = <value>"`
  - WinGame displays rings in a dedicated info box alongside coins/jackpot/high score
  - Ring count also included in simulate data for UI testing
